## 1. 핵심 개념 가이드 (용어 설명)

ABSA 분석에 사용된 주요 개념을 쉽게 풀어서 설명한다.

### ① ABSA란 무엇인가?

**ABSA (Aspect-Based Sentiment Analysis, 속성별 감성 분석)**

일반적인 감성 분석은 리뷰 전체를 "긍정" 또는 "부정"으로만 분류한다. 하지만 ABSA는 **리뷰 안에 담긴 여러 속성(Aspect)** 을 찾아내고, **각 속성마다 감성**을 따로 분석한다.

```
예시 리뷰: "발림성은 좋은데 향이 너무 강해요"

일반 감성 분석:
  → "긍정" 또는 "부정"? (애매함)

ABSA:
  → 사용감/성능: 긍정 (발림성 좋음)
  → 재질/냄새: 부정 (향이 강함)
```

> **왜 ABSA가 필요한가?**
> 단순히 "이 제품은 좋다/나쁘다"를 넘어서, **"어떤 부분이 좋고 어떤 부분이 나쁜지"** 를 구체적으로 파악해야 개선 방향을 찾을 수 있다.

### ② Aspect(속성)이란?

리뷰에서 평가하는 **구체적인 특성**이다. 이 프로젝트에서는 8개 aspect를 정의했다:

| Aspect                | 설명                   | 예시                                |
| --------------------- | ---------------------- | ----------------------------------- |
| **사용감/성능** | 발림성, 지속력, 커버력 | "발림성이 좋아요", "끈적여요"       |
| **색상/발색**   | 색상 만족도, 발색력    | "발색이 좋아요", "색이 안 나와요"   |
| **가격/가성비** | 가격 대비 만족도       | "가성비 좋아요", "비싸요"           |
| **배송/포장**   | 배송 속도, 포장 상태   | "배송 빠르고 포장 꼼꼼"             |
| **디자인**      | 외관, 패키지 디자인    | "예쁘다", "촌스럽다"                |
| **재질/냄새**   | 텍스처, 향             | "촉촉해요", "냄새가 이상해요"       |
| **용량/휴대**   | 용량 만족도, 휴대성    | "양이 적어요", "들고 다니기 좋아요" |
| **재구매**      | 재구매 의사            | "재구매할게요", "다시 안 사요"      |

> **제거된 3개:** 미분류(후처리로 전환), CS/응대(샘플 부족), 품질/퀄리티(경계 모호)

### ③ Sentiment(감성)이란?

각 aspect에 대한 **평가**이다. 4가지로 분류한다:

| 감성               | 의미                                    | 예시            |
| ------------------ | --------------------------------------- | --------------- |
| **none**     | 이 리뷰에서 해당 aspect가 언급되지 않음 | (언급 없음)     |
| **positive** | 긍정적으로 언급됨                       | "발림성 좋아요" |
| **neutral**  | 중립적으로 언급됨                       | "보통이에요"    |
| **negative** | 부정적으로 언급됨                       | "끈적거려요"    |

### ④ 머신러닝 관련 용어

**Multi-task Learning (다중 작업 학습)**

하나의 모델이 여러 작업을 **동시에** 학습하는 방법이다.

```
이 프로젝트에서:
  작업 1: 리뷰 전체의 감성 (긍정/중립/부정)
  작업 2: 8개 aspect 각각의 감성 (none/긍정/중립/부정)
```

> **왜 Multi-task?**
> "리뷰가 전반적으로 긍정"이라는 정보가 "각 aspect의 감성"을 맞추는 데 도움이 되기 때문이다. 두 작업을 함께 학습하면 성능이 올라간다.

**Class Weight (클래스 가중치)**

데이터가 불균형할 때(예: 긍정 9,000개 vs 중립 100개), 적은 클래스에 **높은 가중치**를 줘서 균형을 맞춘다.

```
샘플 수:    긍정 9,000 / 중립 100 / 부정 1,000
가중치:     긍정 0.5  / 중립 8.0  / 부정 1.5

→ 모델이 "전부 긍정으로 찍기"를 하면 오히려 손해가 되도록 유도
```

**Threshold (임계값)**

모델이 출력하는 **확률**을 보고 최종 판정하는 기준이다.

```
모델 출력: P(none)=0.40, P(긍정)=0.35, P(중립)=0.15, P(부정)=0.10

Threshold = 0.50 일 때:
  P(none) = 0.40 < 0.50(기준) → none이 아니다!
  → non-none 중 최대: 긍정(0.35) → 최종 예측: 긍정

Threshold = 0.30 일 때:
  P(none) = 0.40 >= 0.30(기준) → none이다!
  → 최종 예측: none (이 aspect 언급 안 됨)
```

> **왜 Threshold를 조정하나?**
> Aspect마다 언급 빈도가 다르다. "사용감"은 50%에서 언급되지만 "디자인"은 2%에서만 언급되므로, 각각 다른 기준을 적용해야 정확하다.

**Precision vs Recall**

| 지표                | 의미                                  | 비유           |
| ------------------- | ------------------------------------- | -------------- |
| **Precision** | "언급됐다"고 했을 때 실제로 맞을 확률 | 정밀도, 순도   |
| **Recall**    | 실제 언급된 것 중 잡아낸 비율         | 재현율, 포착률 |

```
예시: 실제 언급 100건 / 모델 예측 120건 (그 중 80건이 맞음)

Precision = 80 / 120 = 0.67 (예측 120건 중 67%가 정답)
Recall    = 80 / 100 = 0.80 (실제 100건 중 80%를 잡음)
```

> **Trade-off:** Precision을 높이면(보수적 예측) Recall이 낮아지고(놓침 증가), Recall을 높이면(적극적 예측) Precision이 낮아진다(오탐 증가).

### ⑤ 데이터 포맷

**Long Format vs Wide Format**

```
Long Format (1행 = 1개 aspect):
  review_id | aspect       | aspect_sentiment
  R001      | 사용감/성능  | positive
  R001      | 재질/냄새    | negative
  R002      | 가격/가성비  | positive

Wide Format (1행 = 1개 리뷰, aspect가 컬럼):
  review_id | label_사용감_성능 | label_재질_냄새 | label_가격_가성비
  R001      | 1 (positive)      | 3 (negative)    | 0 (none)
  R002      | 0 (none)          | 0 (none)        | 1 (positive)
```

> **왜 두 가지 포맷?**
> Long은 GPT 라벨링 결과를 그대로 보존하기 좋고, Wide는 머신러닝 모델 학습에 적합하다.

## 2. 왜 ABSA를 하는가? (분석 동기)

### 비즈니스 질문

다이소 뷰티의 **연착륙 제품**을 발굴하려면, "어떤 제품이 왜 오래 사랑받는가?"를 알아야 한다.

단순 평점만으로는 부족하다:

- 평점 5점 제품도 "배송은 좋은데 발림성이 별로"일 수 있음
- "가성비는 좋은데 향이 싫다"는 제품은 개선 여지가 있음

### ABSA로 할 수 있는 것

| 질문                              | ABSA 답변                                    |
| --------------------------------- | -------------------------------------------- |
| **강점이 뭔가?**            | "사용감/성능 긍정 78%, 가격/가성비 긍정 98%" |
| **약점이 뭔가?**            | "재질/냄새 부정 30%, 배송/포장 부정 24%"     |
| **개선해야 할 것은?**       | "용량/휴대 부정 21% → 용량 늘리기"          |
| **어떤 제품이 연착륙인가?** | "사용감 긍정 + 재구매 긍정 + 부정 낮음"      |

### 데이터 규모

| 항목               | 수치                      |
| ------------------ | ------------------------- |
| 총 리뷰            | 323,114건                 |
| 총 상품            | 937개                     |
| 라벨링 샘플        | 20,000건 (GPT-4o)         |
| 골든셋 (직접 검수) | 884건                     |
| 최종 학습 데이터   | 13,749건                  |
| 추론 시간          | 66.1분 (81.4 reviews/sec) |
| 총 비용            | ~$99 (GPT-4o Batch API)   |

## 4. Step 1: 데이터 준비 — 샘플링과 라벨링

### 4.1 무엇을 하는가?

전체 323,114건의 리뷰를 모두 사람이 라벨링하는 것은 불가능하다. 대표성 있는 **20,000건**을 선별하여 GPT-4o로 라벨링한다.

### 4.2 층화 샘플링

**"층화"란?**

> 전체 데이터의 분포(카테고리, 평점 등)를 그대로 유지하면서 샘플을 뽑는 방법이다.

```
전체 리뷰:
  스킨케어 70% / 메이크업 28% / 맨케어 2%
  평점 5점 80% / 평점 3점 10% / 평점 1점 10%

층화 샘플링:
  20,000건에서도 위 비율을 동일하게 유지
  → 대표성 확보
```

**3단계 층화:**

1. **대분류** (스킨케어/메이크업/맨케어) 비율 유지
2. **소분류** (기초스킨케어/립메이크업 등) 비율 유지
3. **평점** (1~5점) 비율 유지

### 4.3 결과

| 항목          | 값                                        |
| ------------- | ----------------------------------------- |
| 샘플 수       | 20,000건                                  |
| 카테고리 분포 | 기초스킨케어 38.6%, 립메이크업 11.7%, ... |
| 평점 분포     | 5점 44.0%, 3점 30.0%, 1점 9.6%, ...       |

## 5. Step 2: GPT-4o로 초기 라벨 생성

### 5.1 왜 GPT를 사용하는가?

사람이 직접 20,000건을 라벨링하면:

- 시간: 1건당 3분 × 20,000 = **1,000시간** (125일)
- 비용: 시간당 1만원 × 1,000시간 = **1,000만원**

GPT-4o Batch API 사용:

- 시간: **3시간** (병렬 처리)
- 비용: **$95.70** (~13만원)

→ **시간 99.7% 단축, 비용 99% 절감**

### 5.2 GPT-4o vs GPT-4o-mini 비교

골든셋 430건으로 두 모델을 비교했다:

| 모델             | 비용 (20K 기준)  | Aspect 정확도   | Sentiment 정확도 | Both 정확도     |
| ---------------- | ---------------- | --------------- | ---------------- | --------------- |
| **GPT-4o** | **$95.70** | **67.6%** | **72.1%**  | **58.3%** |
| GPT-4o-mini      | $2.87            | 51.2%           | 67.0%            | 43.5%           |

> **결론:** GPT-4o가 15%p 더 정확. B2B 입점 제안 목적상 정확도가 중요 → **GPT-4o 채택**

### 5.3 프롬프트 개선 과정

**1차 라벨링 (GPT-4o-mini):**

- 오류율 높음 (골든셋 대비 43.5%)
- 주요 오류: "색상/발색 ↔ 사용감/성능" 혼동 (26회, 93%)

**개선 사항:**

```
1. Aspect 혼동 방지 규칙 추가
   - "발색, 색, 컬러" → 무조건 색상/발색
   - "발림, 지속, 커버" → 사용감/성능
   - "발색" ≠ "발림" 명시

2. 평점 기반 Sentiment 보조 판단
   - 1~2점: negative 가능성 높음
   - 4~5점: positive 가능성 높음

3. 다이소 특화 규칙
   - "올리브영 XX랑 똑같다" → 가격/가성비 positive
   - "구하기 힘들어서 짜증 (5점)" → 재구매 positive
```

### 5.4 Batch API 실행

```
총 20,000건을 2개 API 키로 병렬 처리:
  키 1: 13,150건 ($62.92)
  키 2:  6,850건 ($32.78)
```

## 6. Step 3: 모델 설계 — KcELECTRA + Multi-task

### 6.1 왜 KcELECTRA인가?

| 모델 후보           | 장점                                     | 단점                 | 선택         |
| ------------------- | ---------------------------------------- | -------------------- | ------------ |
| KoBERT              | 한국어 사전학습                          | 느림, 성능 보통      | ✗           |
| **KcELECTRA** | **4배 빠름, 적은 데이터로도 우수** | -                    | **✓** |
| GPT 계열            | 최고 성능                                | 비용 높음, 추론 느림 | ✗           |

> KcELECTRA-base (`beomi/KcELECTRA-base`, 768차원)를 사용하여 한국어 리뷰에 최적화하면서도 빠른 학습·추론이 가능하다.

### 6.2 모델 아키텍처 (Option A)

```
리뷰 텍스트: "발림성 좋고 가격도 착해요"
      ↓
┌─────────────────────────────────────┐
│  KcELECTRA Encoder (768-dim)        │
│  한국어 사전학습 모델                  │
└──────────┬──────────────────────────┘
           │ [CLS] 토큰 표현 (768차원)
    ┌──────┴──────┐
    ↓             ↓
┌──────────┐  ┌────────────────┐
│Sentiment │  │Aspect-Sentiment│  ← 메인 태스크
│  Head    │  │     Head       │
│768 → 3   │  │768 → 32 → [8,4]│
└──────────┘  └────────────────┘
    ↓             ↓
 [B, 3]        [B, 8, 4]
neg/neu/pos   aspect별 none/pos/neu/neg
(보조 태스크)  (메인 태스크)
```

**핵심 설계 결정:**

1. **Aspect별 4-class 통합:**
   - **각 aspect별로 감성을 직접 예측** (none/긍정/중립/부정)

```
예시: "발림성은 좋은데 향이 별로"
  → 사용감/성능 = positive
  → 재질/냄새 = negative
  → 나머지 = none
```

2. **Masked CrossEntropyLoss:**
   - `aspect_mask=1`인 셀만 loss 계산에 포함
   - 불확실한 라벨(mask=0)은 학습에서 제외

```python
# CE(reduction='none') → [B*8] → reshape [B, 8]
per_cell = F.cross_entropy(logits, labels, weight=class_weights, reduction="none")
per_cell = per_cell.view(B, 8)

# mask=1인 셀만 합산
masked_loss = (per_cell * aspect_mask).sum() / aspect_mask.sum()
```

3. **Class Weight (역빈도 가중치):**
   - none이 90%를 차지 → none 가중치 낮춤
   - neutral이 3%만 차지 → neutral 가중치 높임

### 6.3 평가 지표

| 지표                             | 설명                                                               |
| -------------------------------- | ------------------------------------------------------------------ |
| **Detection F1**           | "이 리뷰에서 aspect가 언급되었는가?" (none vs non-none)            |
| **Mentioned Sentiment F1** | "실제로 언급된 aspect의 감성을 맞추는가?" (pos/neu/neg, none 제외) |
| **Aspect별 F1**            | 각 aspect의 4-class 분류 성능                                      |

> **핵심 지표:** Detection Precision ≥ 0.65 (오탐 억제) + Mentioned Sentiment F1 (감성 정확도)

### 8.2 Negative Sampling (디자인)

**문제:** 디자인 aspect는 mask=0(미라벨)이 약 11,000건 → "none이 무엇인지" 학습할 데이터 부족

```

  (2) 디자인 키워드 24개 중 0개 포함이고
  (3) per-review cap (1개) 미도달

보강 결과 (디자인):

  label=0 (none): 2,345 → 4,845 (+2,500건)
```

### 8.3 F0.5 Threshold Metric

**F1 (기존):**

```
F1 = 2 × P × R / (P + R)
     (Precision과 Recall을 동등하게 평가)
```

**F0.5 (신규):**

```
F0.5 = 1.25 × P × R / (0.25 × P + R)
       (Precision을 Recall보다 4배 중시)
```

> **왜 F0.5?**
> 오탐(FP)이 비즈니스적으로 더 위험하다. "실제로 없는 것을 있다고 세면" 분석 결과가 왜곡되므로, Precision을 우선한다.

### 9.1 Stage 2 → Stage 3A: 무엇이 바뀌었는가?

### 9.2 ① Design Rule Override (디자인 규칙 전환)

**문제:** 디자인 Precision 0.17 (오탐이 FP의 주범)

**해결:** 모델의 디자인 예측을 완전히 무시하고, **키워드 규칙**으로 대체

```
디자인 키워드 (우선순위 순):
  Tier 1 (positive): 예쁘, 이쁘, 귀엽, 깔끔, 고급, 앙증, 세련
  Tier 2 (negative): 촌스럽, 투박, 칙칙, 유치, 조잡, 싸구려
  Tier 3 (구조물+수식어): (디자인|패키지|용기|뚜껑) + (좋|이쁘|별로|아쉽)

로직:
  if Tier1 매칭 → 디자인=positive
  elif Tier2 매칭 → 디자인=negative
  elif Tier3 매칭 → 조합 감성 판단
  else → 디자인=none
```

**장점:** Precision 극적 개선 (0.17 → 0.50), 즉시 적용 가능
**단점:** Recall 한계 (간접 표현 누락), 유지보수 부담

### 9.3 ② 사용감/성능 Threshold 보정

**문제:** 사용감/성능 +23pp 과다검출 (GT 57% vs Pred 80%)

**발견:** Threshold의 의미가 **직관과 반대**

```
직관:
  "threshold를 높이면 기준이 높아지니 검출이 줄어들겠지"

실제:
  threshold를 높이면 → P(none)이 기준 못 넘음 → 검출 증가!
  threshold를 낮추면 → P(none)이 쉽게 기준 넘음 → 검출 감소!

Threshold 작동 방식:
  if P(none) >= threshold:
      predict = none  (미언급)
  else:
      predict = argmax(pos, neu, neg)  (언급됨)
```

**Golden Dev Grid Search:**

| threshold      | Pred 언급%      | GT 언급%        | ±pp           | Precision       | 판정                 |
| -------------- | --------------- | --------------- | -------------- | --------------- | -------------------- |
| 0.01           | 42.2%           | 55.8%           | -13.6          | 0.811           | 과소 검출            |
| **0.03** | **52.2%** | **55.8%** | **-3.6** | **0.773** | **최적**       |
| 0.05           | 67.5%           | 55.8%           | +11.7          | 0.689           | 소폭 과다            |
| 0.10           | 80.2%           | 55.8%           | +24.4          | 0.644           | 심각 과다 ← Stage 2 |

**해결:** threshold 0.10 → **0.03** (낮춤)으로 과다검출 억제

### 9.4 ③ Polar Threshold 조정

**Polar threshold란?**

> "긍정/부정 중 어느 쪽이든 확률이 polar threshold 미만이면 neutral로 전환"하는 안전장치

```
Polar threshold = 0.55 작동 방식:

Step 1: P(none) >= t_none → none (미언급)
Step 2: max(P(pos), P(neg)) < 0.55 → neutral (확신 부족)
Step 3: else → argmax(pos, neg)

예시:
  P(none)=0.20, P(pos)=0.45, P(neg)=0.30
  Step 1: 0.20 < threshold → 언급됨
  Step 2: max(0.45, 0.30)=0.45 < 0.55 → neutral!
```

**Stage 2 문제:** polar_threshold=0.55로 neutral 과잉예측 (155셀 vs GT 58셀)

**Stage 3A 해결:** polar_threshold 0.55 → **0.50** (낮춤)으로 neutral 기준 엄격화

### 9.5 Golden Test 최종 평가

**결과:**

| 지표                              | Stage 2 | Stage 3A         | 변화              | GO 기준 | 판정         |
| --------------------------------- | ------- | ---------------- | ----------------- | ------- | ------------ |
| **Detection Precision**     | 0.6079  | **0.6708** | **+6.3pp**  | ≥ 0.65 | **GO** |
| Detection Recall                  | 0.6993  | 0.6173           | -8.2pp            | -       | -            |

| **사용감 ±pp**             | +23.0   | **-6.5**   | **-29.5pp** | ≤ ±15 | **GO** |
| **7 aspect 평균 Precision** | -       | **0.6765** | -                 | ≥ 0.60 | **GO** |

**판정:** 4개 GO 기준 모두 충족 → **GO — 30만 건 추론 진행**

**Trade-off 분석:**

얻은 것:

- Detection Precision +6.3pp → 목표 0.65 달성
- 사용감 과다검출 해소 (+23pp → -6.5pp)

잃은 것:

**판단:** 30만 건 분석에서 **카운트 정확도(Precision)** 가 **민감도(Recall)** 보다 중요. 오탐이 집계 왜곡을 일으키므로, 이 trade-off는 합리적.

## 13. Step 10: 최종 결과 — S4B + Regex Gate

### 13.1 Keyword Gate 정규식 전환

**동기:** 기존 Substring 매칭의 오탐 문제

```
예시: "싸" substring 매칭
  "가격이 싸요" ✓ (정답)
  "잽싸게 흡수" ✗ (오탐 — '싸' 포함)
  "포싸패치" ✗ (오탐)
  "싸움터" ✗ (오탐)
```

**정규식 Gate 설계 원칙:**

| 원칙                     | 적용 예시                                  |
| ------------------------ | ------------------------------------------ |
| **활용형만 매칭**  | `싸` → `싸[다고게서니요네]`           |
| **부정 전방탐색**  | `깔끔` → `깔끔(?!\s*(?:하게\s*)?(?:흡수 |
| **문맥 한정**      | `양` → `(?<![가-힣])양[이은도만]\s`   |
| **오탐 패턴 제외** | `향` → `향(?!상                         |
| **복합 구조물만**  | `팁` → `(?:실리콘                       |

**Gate 대상 4개 Aspect:**

| Aspect                | Substring 키워드 수 | Regex 패턴 수 | 주요 변경                                             |
| --------------------- | :-----------------: | :------------: | ----------------------------------------------------- |
| 디자인                |        19개        |      14개      | `깔끔`, `고급`, `팁` — 사용감 맥락 제외        |
| **가격/가성비** |   **14개**   | **12개** | **`싸` → 활용형만, `싸게` → 잽싸게 제외** |
| 재질/냄새             |        13개        |      16개      | `향` → 향상/향수 제외, `쏘` → 활용형만          |
| 용량/휴대             |        16개        |      18개      | `양` → 독립 용량 맥락만, `g` → 숫자+g만         |

### 14.5 교훈

**① 데이터 품질이 모델 천장을 결정한다**

> GPT-4o 자동 라벨과 골든셋 사람 라벨 사이의 gap이 모델 성능의 한계였다. 골든셋 파인튜닝(Stage 5)으로 이 gap을 줄였다.

**② 클래스 불균형은 데이터 보강으로 해결**

> Neutral이 2.5%만 차지 → class weight 8.06으로 보상 → 한계 → 750건 증강 → class weight 6.82로 하락 → Neutral Recall 0.28 달성

**③ 후처리가 재학습보다 빠르고 안전할 수 있다**

> Stage 3A: 재학습 없이 후처리만으로 Precision 0.61 → 0.67 달성. 1-2일 작업으로 GO 판정.

**④ Trade-off는 비즈니스 목표에 맞춰 선택**

> Precision vs Recall: 30만 건 집계 분석에서는 Precision이 더 중요 → 보수적 과소검출(-9.6pp) 선택

**⑤ 다중 방법론 검증이 신뢰도를 높인다**

> 4가지 Stage(1/2/3A/4-5), 3가지 전략(A/B/C), 2가지 Gate(Substring/Regex) 교차 검증으로 최적 조합 도출
